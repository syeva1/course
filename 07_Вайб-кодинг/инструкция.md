# Инструкция — Модуль 07. Вайб-кодинг

## Шаг 1. Что такое вайб-кодинг

Вайб-кодинг — это создание продуктов через AI без классического программирования. Ты описываешь что хочешь, AI пишет код, ты проверяешь и уточняешь.

Это не игрушка. Люди создают реальные продукты: лендинги для бизнеса, калькуляторы, парсеры данных, Chrome-расширения, дашборды с метриками. За вечер можно сделать то, что раньше занимало неделю.

Что можно создать:
- **Лендинг** — страница для рекламы услуги/продукта
- **Калькулятор стоимости** — клиент вводит параметры, видит цену
- **Парсер** — скрипт, который собирает данные с сайтов
- **Chrome-расширение** — плагин для браузера
- **Дашборд** — панель с метриками и графиками
- **Квиз** — интерактивный опросник для лидогенерации
- **Портфолио** — персональный сайт с работами

Результат: Ты понимаешь что можно создать и выбираешь свой первый продукт.

## Шаг 2. Создай README.md — главный файл проекта

README.md — это первый файл, который Cursor прочитает. Чем лучше README, тем лучше результат.

1. Создай новую папку для проекта (например, `мой-лендинг`)
2. Открой в Cursor: File → Open Folder
3. Создай файл `README.md`
4. Заполни по шаблону:

```markdown
# Название проекта

Лендинг для HR-агентства по подбору бизнес-ассистентов.

## Для кого
Предприниматели, которые тратят время на рутину и ищут ассистента.

## Функции
- Герой-секция с заголовком и кнопкой "Оставить заявку"
- Блок с 3 преимуществами
- Отзывы клиентов (3 карточки)
- Форма заявки (имя, телефон, email)
- Футер с контактами и соцсетями

## Дизайн
- Цвета: красный #E63229 (акцент), чёрный #1A1A1A (текст), белый (фон)
- Шрифт: Montserrat
- Стиль: профессиональный, современный, минималистичный
- Адаптивный (мобилка + десктоп)

## Технологии
HTML, CSS, JavaScript. Без фреймворков.
```

5. Сохрани файл

Зачем так подробно: AI не читает мысли. Чем больше деталей в README, тем точнее результат с первого раза. Цвета, шрифт, секции, технологии — всё важно.

Готовый шаблон — в `материалы/шаблон_readme.md`.

Результат: README.md описывает продукт так, что AI может создать его без дополнительных вопросов.

## Шаг 3. Plan Mode — составь план

Не проси AI сразу делать весь проект. Сначала план.

1. Открой Composer (Cmd+I)
2. Переключи в режим Plan (выпадающий список внизу)
3. Напиши:

```
Прочитай README.md. Составь пошаговый план реализации этого проекта. 
Для каждого шага укажи: что создать, какие файлы, ожидаемый результат.
Шаги должны быть маленькими — каждый занимает 1-2 минуты.
```

4. AI покажет план. Прочитай его.
5. Если что-то не так — напиши: «Измени шаг 3, я хочу сначала стили, потом контент»
6. Когда план устраивает — сохрани его в файл `plan.md`

Зачем plan.md: если Cursor потеряет контекст (новый чат, перезагрузка), ты всегда можешь сказать «продолжи по plan.md».

Результат: У тебя есть план из 8-12 шагов. Каждый шаг конкретный и маленький.

## Шаг 4. Первый шаг — базовая структура

1. Переключи Composer в Agent Mode
2. Напиши:

```
Выполни шаг 1 из plan.md. Создай базовую HTML-структуру с подключением стилей.
```

3. AI создаст файлы (index.html, css/styles.css)
4. Нажми Accept All
5. Открой index.html в браузере:
   - Правый клик на файле → Reveal in Finder/Explorer
   - Двойной клик на файле в проводнике
   - Или если установил Live Server — правый клик → Open with Live Server

6. Посмотри результат в браузере

Результат: Базовая страница открывается в браузере.

## Шаг 5. Поэтапная реализация

Теперь проходишь план шаг за шагом. Для каждого шага:

1. Напиши в Composer: «Выполни шаг N из plan.md»
2. AI создаст/изменит файлы
3. Обнови страницу в браузере (F5 или Ctrl+R)
4. Если результат хороший → commit:
```bash
git add .
git commit -m "Шаг N: добавил секцию услуг"
```
5. Если результат плохой → скажи AI что не так:
```
Секция услуг выглядит кривой. Карточки должны быть в одну линию на десктопе 
и в столбик на мобилке. Между карточками отступ 20px.
```

Главное правило: один запрос = одна функция. Не проси «сделай весь лендинг» — проси «добавь секцию отзывов».

Commit после каждого рабочего шага. Если сломаешь — можешь откатиться:
```bash
git checkout -- .
```

Результат: Продукт растёт шаг за шагом. Каждый шаг закоммичен.

## Шаг 6. Уточнения и итерации

AI не сделает идеально с первого раза. Это нормально. Уточняй:

Плохой уточняющий промпт:
```
Сделай лучше
```

Хороший уточняющий промпт:
```
Кнопка "Оставить заявку" слишком маленькая на мобилке. 
Сделай её шириной 100%, высотой 56px, размер текста 18px. 
Скругление углов 12px. Красный фон #E63229, белый текст.
```

Чем конкретнее описываешь проблему, тем точнее AI исправит. Описывай:
- Что именно не так (кнопка маленькая, текст не виден, отступ большой)
- Какой результат хочешь (ширина 100%, цвет красный, шрифт 18px)
- Где именно (на мобилке, в секции отзывов, в футере)

Результат: Ты умеешь давать конкретные уточнения.

## Шаг 7. Адаптив для мобилки

Любой продукт должен нормально выглядеть на телефоне.

Как проверить:
1. Открой страницу в Chrome
2. Нажми F12 (DevTools)
3. Нажми иконку телефона (Toggle device toolbar) или Ctrl+Shift+M
4. Выбери устройство (iPhone 14, Samsung Galaxy — любое)
5. Проверь: текст читается? Кнопки нажимаются? Ничего не обрезано?

Если что-то кривое — попроси AI:
```
На мобильном (ширина 375px) карточки услуг наезжают друг на друга. 
Сделай их в одну колонку на экранах меньше 768px. 
Текст заголовка: 24px на мобилке (вместо 48px на десктопе).
```

Результат: Продукт одинаково хорошо выглядит на телефоне и компьютере.

## Шаг 8. Финальная проверка

Перед деплоем проверь:

1. Открой на телефоне (скинь себе ссылку или через Live Server в локальной сети)
2. Проверь все кнопки и ссылки — работают?
3. Проверь форму — данные отправляются?
4. Проверь скорость — страница грузится быстро?
5. Попроси кого-то посмотреть — свежий взгляд находит баги

Что часто забывают:
- Favicon (иконка вкладки)
- Мета-теги (title, description)
- Ссылки на соцсети
- Адаптив для планшета (768-1024px)

Попроси AI проверить:
```
Проверь index.html на типичные ошибки: мета-теги, favicon, alt у картинок, адаптив. Что я забыл?
```

Результат: Продукт проверен и готов к публикации.

## Шаг 9. Деплой на GitHub Pages (бесплатно)

GitHub Pages — бесплатный хостинг для HTML/CSS/JS сайтов.

1. Залей проект на GitHub (если ещё нет):
```bash
git add .
git commit -m "Готовый проект"
git push
```

2. На github.com открой репозиторий
3. Зайди в Settings → Pages
4. Source: Deploy from a branch
5. Branch: main, папка: / (root)
6. Нажми Save
7. Через 1-2 минуты сайт будет доступен по адресу: `https://твой-username.github.io/название-репо/`

Подробный гайд — в `материалы/деплой_github_pages.md`.

Результат: Твой продукт доступен в интернете по ссылке.

## Шаг 10. Декомпозиция больших проектов

Для простого лендинга шагов 1-9 достаточно. Но для большого проекта нужна декомпозиция.

Правила:
1. **Один файл plan.md** — весь план в одном месте
2. **Файл progress.md** — что сделано, что нет (обновляй после каждого шага)
3. **Максимум 1 файл за шаг** — если AI меняет 5 файлов за раз, разбей на 5 шагов
4. **Commit после каждого шага** — возможность откатиться
5. **Проверка после каждого шага** — не копи долг

Шаблон progress.md:
```markdown
## Прогресс

- [x] Шаг 1: Базовая структура HTML
- [x] Шаг 2: Стили (цвета, шрифты)
- [ ] Шаг 3: Секция героя
- [ ] Шаг 4: Секция услуг
...
```

Подробнее — в `материалы/большие_проекты_декомпозиция.md`.

Результат: Ты умеешь вести большие проекты с контролем прогресса.
