# Slash-команды в Cursor (/commands)

> Многоразовые воркфлоу для повторяющихся задач.
> Источник: официальный гайд Cursor

---

## Что такое Commands

Команды — это готовые промпты, которые запускаются через `/` в поле ввода агента.

Хранятся в `.cursor/commands/` как Markdown-файлы.

---

## Как создать

### 1. Создай папку
```
.cursor/commands/
```

### 2. Создай Markdown-файл

`.cursor/commands/pr.md`:
```markdown
Создай Pull Request для текущих изменений.

1. Посмотри staged и unstaged изменения через `git diff`
2. Напиши понятный commit message по сути изменений
3. Закоммить и запушь в текущую ветку
4. Используй `gh pr create` для создания PR
5. Верни URL созданного PR
```

### 3. Используй
Набери `/pr` в поле ввода → Cursor выполнит всё автоматически.

## Готовые команды

### /pr — Pull Request
```markdown
Создай Pull Request:
1. `git diff` — посмотри изменения
2. Напиши commit message
3. `git commit && git push`
4. `gh pr create` с заголовком и описанием
5. Верни URL
```

### /fix-issue — Исправить баг из GitHub
```markdown
Исправь issue #[number]:
1. `gh issue view [number]` — прочитай описание
2. Найди связанный код
3. Реализуй исправление
4. Запусти тесты
5. Открой PR с ссылкой на issue
```

### /review — Ревью кода
```markdown
Проведи ревью текущих изменений:
1. Запусти линтер
2. Проверь типы
3. Проверь на типичные проблемы:
   - Утечки секретов
   - Неиспользуемые импорты
   - Дублирование кода
4. Резюме: на что обратить внимание
```

### /update-deps — Обновить зависимости
```markdown
Обнови зависимости:
1. Проверь устаревшие пакеты
2. Обновляй по одному
3. После каждого обновления запусти тесты
4. Если тесты упали — откати и пропусти
5. Отчёт: что обновлено, что пропущено
```

### /deploy — Деплой
```markdown
Задеплой текущую версию:
1. Запусти все тесты
2. Если тесты прошли → `git push`
3. Проверь, что CI/CD отработал
4. Верни статус деплоя
```

### /docs — Обновить документацию
```markdown
Обнови документацию:
1. Найди все изменённые файлы с последнего коммита
2. Проверь, есть ли для них документация
3. Обнови/создай docstrings
4. Обнови README если нужно
```

---

## Советы

- Коммить команды в Git — **вся команда** сможет ими пользоваться
- Одна команда = один файл
- Используй конкретные инструкции (не «проверь всё», а «проверь типы, линтер, тесты»)
- Тестируй на реальных задачах → дорабатывай формулировки
