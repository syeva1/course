# Инструкция — Модуль 04. Агенты и скиллы

## Шаг 1. Что такое .mdc и зачем это нужно

В Модуле 02 мы создали .cursorrules — файл, который AI читает при каждом запросе. Но .cursorrules — это один файл на весь проект.

.mdc файлы (Markdown Cursor) — это правила, которые подключаются по ситуации. Каждый .mdc — это отдельный «сотрудник» со своей ролью.

Например:
- `копирайтер.mdc` — пишет тексты в нужном тоне
- `аналитик.mdc` — анализирует данные по шаблону
- `рекрутер.mdc` — составляет вакансии и оценивает резюме

Разница с .cursorrules:
- .cursorrules — общий контекст, работает ВСЕГДА
- .mdc — специализированный контекст, работает когда нужен

Где лежат: в папке `.cursor/rules/` внутри проекта.

Результат: Ты понимаешь зачем нужны .mdc и чем они отличаются от .cursorrules.

## Шаг 2. Создай первый .mdc файл

1. Открой свой проект в Cursor
2. Создай папку `.cursor/rules/` (если её нет):
   - В терминале: `mkdir -p .cursor/rules`
   - Или через File Explorer: правый клик → New Folder
3. Создай файл `.cursor/rules/копирайтер.mdc`
4. Вставь содержимое:

```
---
description: Копирайтер для бизнес-текстов
globs: ["**/*.md", "**/*.txt", "**/*.html"]
alwaysApply: false
---

## Роль
Ты — копирайтер агентства по подбору бизнес-ассистентов.

## Тон
Профессиональный, но дружелюбный. Без канцелярита. Пишешь как умный человек, а не как робот.

## Целевая аудитория
Предприниматели, владельцы малого и среднего бизнеса. Ценят время, не любят воду.

## Правила
- Заголовки: короткие, с выгодой
- Абзацы: максимум 3 предложения
- Один текст = одна главная мысль
- Не использовать: «уникальный», «инновационный», «лидер рынка»
- Call to action в конце каждого текста
```

Разбор структуры:
- `description` — когда AI подключит это правило (описание для себя)
- `globs` — для каких файлов правило работает (*.md = все маркдаун-файлы)
- `alwaysApply: false` — правило подключается автоматически по глобу, но не навязывается
- Текст после `---` — сами инструкции для AI

5. Сохрани файл

Проверка: открой или создай файл `текст_для_сайта.md` и в Chat (Cmd+L) напиши:
```
Напиши текст для главной страницы сайта
```

AI должен писать в стиле, описанном в .mdc (профессионально, без воды, с call to action).

Результат: Первый .mdc файл создан и работает.

## Шаг 3. Создай агента-аналитика

Второй агент — для другой задачи.

1. Создай файл `.cursor/rules/аналитик.mdc`
2. Содержимое:

```
---
description: Аналитик бизнес-данных
globs: ["**/*.csv", "**/*.xlsx", "**/*.json"]
alwaysApply: false
---

## Роль
Ты — бизнес-аналитик. Анализируешь данные и даёшь выводы.

## Формат ответа
1. Главный вывод (1-2 предложения)
2. Ключевые цифры (3-5 пунктов)
3. Что делать дальше (1-2 рекомендации)

## Правила
- Всегда считай в конкретных числах, не в процентах без базы
- Если данных мало — скажи прямо, не додумывай
- Сравнивай с прошлым периодом если есть данные
- Формат чисел: 1 234 567 (с пробелами)
```

3. Создай тестовый файл `продажи.csv`:
```
Месяц,Выручка,Расходы
Январь,500000,320000
Февраль,620000,310000
Март,580000,340000
```

4. В Chat напиши:
```
@File продажи.csv Проанализируй данные
```

Результат: AI отвечает в формате аналитика — вывод, цифры, рекомендации.

## Шаг 4. Разница между промптами и скиллами

Промпт — одноразовый. Написал, получил ответ, забыл. В следующий раз пишешь заново.

Скилл (.mdc) — многоразовый. Настроил один раз, работает при каждом запросе автоматически.

Пример:
- Промпт: «Напиши текст для сайта в профессиональном тоне, для предпринимателей, без воды, с call to action» — каждый раз писать заново
- Скилл: файл `копирайтер.mdc` — AI сам подключает его когда ты работаешь с текстовыми файлами

Вывод: Чем больше у тебя .mdc файлов, тем меньше ты пишешь в промптах. AI уже знает как работать.

В папке `материалы/готовые_mdc/` лежат готовые .mdc файлы. Скопируй нужные в свой `.cursor/rules/`.

Результат: Ты понимаешь почему скиллы > промпты.

## Шаг 5. Выбор модели

В Cursor можно переключать модели AI. Внизу окна Composer есть выпадающий список с моделями.

Какие модели есть (2026):
- **Claude Sonnet** — хорош для творческих задач, текстов, анализа
- **GPT-4o** — хорош для кода, точных инструкций
- **Gemini Pro** — хорош для работы с большими файлами (длинный контекст)
- **Auto** — Cursor сам выбирает модель под задачу

Главное правило: не влюбляйся в модель. Они обновляются каждый месяц. То что было лучшим вчера — завтра может быть вторым.

Как проверить какая модель лучше: сайт lmarena.ai — рейтинг моделей на основе слепого голосования.

Как переключить:
1. Открой Composer (Cmd+I) или Chat (Cmd+L)
2. Внизу окна — название текущей модели
3. Кликни → выпадающий список
4. Выбери другую модель

Результат: Ты знаешь какие модели есть и умеешь переключать.

## Шаг 6. Тройной прогон

Для важных задач (текст для клиента, стратегия, ответственный код) используй Тройной прогон:

1. Выполни задачу в одной модели (например, Claude Sonnet)
2. Переключи модель → выполни ту же задачу (например, GPT-4o)
3. Сравни результаты
4. Возьми лучшее из обоих (или попроси третью модель объединить)

Зачем: разные модели по-разному думают. Claude лучше пишет тексты, GPT точнее с кодом, Gemini лучше работает с длинными документами.

Когда использовать: только для важных задач. Для мелочей — одной модели достаточно.

Результат: Ты попробовал одну задачу на двух моделях и видишь разницу.

## Шаг 7. lessons.md — обучение агента

Агент будет ошибаться. Это нормально. Важно записывать ошибки, чтобы он не повторял их.

1. Создай файл `lessons.md` в корне проекта
2. Формат записи:

```
## 2026-03-10

### Проблема
Копирайтер использовал слово «уникальный» в тексте, хотя в правилах это запрещено.

### Решение
Добавил в .mdc: «Перед финальным ответом проверь текст на запрещённые слова.»

### Результат
Больше не использует запрещённые слова.
```

3. Добавь в .cursorrules строку:
```
Перед каждым ответом проверяй lessons.md — не повторяй ошибки из этого файла.
```

Зачем: AI учится на своих ошибках. Но он не помнит прошлые разговоры. Файл lessons.md — это его память.

Результат: У тебя есть система улучшения агентов.

## Шаг 8. Hooks — автоматические проверки

Hooks — это действия, которые выполняются автоматически после каждого ответа AI.

Например: после того как агент написал код — автоматически прогнать тесты.

Как настроить:
1. Создай файл `.cursor/hooks.json` в проекте
2. Пример:

```json
{
  "post-apply": [
    {
      "command": "prettier --write {files}",
      "description": "Форматирую код"
    }
  ]
}
```

Это значит: после каждого Apply (когда AI изменяет файлы) — автоматически запустить Prettier для форматирования.

Подробнее про hooks — в `материалы/hooks_автопроверки.md`.

Результат: Ты понимаешь что такое hooks и как они автоматизируют проверку.

## Шаг 9. Мульти-агентная работа

Для больших проектов можно использовать несколько агентов одновременно.

Простой способ (один Cursor):
- Создай несколько .mdc файлов с разными ролями
- Переключайся между задачами — AI подхватывает нужный .mdc по глобу файла

Продвинутый способ (Git Worktree):
- Создай несколько копий проекта через `git worktree add`
- Открой каждую копию в отдельном окне Cursor
- Каждый Cursor работает как отдельный агент
- Координируй через общую папку с файлами-задачами

Реальный кейс в `материалы/кейс_3_агента_claude_code.md` — как один разработчик за 4 недели собрал SaaS с платежами, используя 3 агента.

Результат: Ты знаешь два способа мульти-агентной работы.
